To handle a 30-page case file within the Groq Free Tier (6,000 TPM), you must move away from "Stuffing" (putting everything in one prompt) and adopt a Hierarchical Retrieval and Summarization strategy.

Since 30 pages of legal text can easily exceed 20,000–30,000 tokens, a single request would be 5x your limit. Here is the architectural blueprint to make this work.

1. The "Map-Reduce" Summarization Pattern
Instead of summarizing the whole file at once, break it into small chunks, summarize them individually (Map), and then combine those summaries (Reduce).

Step 1 (Map): Split the 30 pages into 15 chunks (approx. 2 pages each).

Step 2: Send each chunk to the llama-3.1-8b-instant model with a strict prompt: "Extract only key legal sections and core facts from this segment. Limit output to 150 tokens."

Step 3 (Reduce): Take those 15 mini-summaries (now totaling ~2,250 tokens) and send them to llama-3.3-70b-versatile for the final cohesive summary.

2. Hierarchical RAG (Summary + Chunk)
To find relevant case laws and sections accurately, you should use a Two-Tier Vector Store:

Top Level (The Map): Create a vector index of the summaries of each page.

Bottom Level (The Details): Create a vector index of the raw text chunks.

How the Agent searches:

The Agent first searches the Top Level (summaries) to find which pages are likely to contain the "applicable sections."

Once it identifies Pages 4, 12, and 18, it only retrieves the raw chunks for those specific pages.

This keeps the context window small (e.g., 2,000 tokens instead of 30,000).

4. Bypassing the 429 Error (The "Leaky Bucket")
Since you are on the Free Tier, you must implement a Token Budgeter in your code:

Throttle Requests: Set max_rpm=1 in your Crew initialization.

Truncate Context: In your custom RAG tool, never return more than 1,500 tokens of context. If len(context) > 1500, use a simple Python slice context[:1500] or a local BERT summarizer as we discussed previously.

Avoid "Planning" Mode: In CrewAI, set planning=False. This prevents the "Manager" agent from making extra high-token "Strategy" calls to Groq.

Summary of the "Legal Deep Dive" Workflow:
User uploads 30 pages.

System-side: Automatically chunks and creates a "Legal Index" (a list of what's on each page).

Researcher Agent: Looks at the Index first (low token cost).

Researcher Agent: Pulls only the 2-3 most relevant full pages for deep analysis.

Advisor Agent: Finalizes the answer.

By moving from "Search all text" to "Search the index, then read the specific page," you reduce your token usage by 90%, making 30-page files perfectly possible on the Groq Free Tier.
